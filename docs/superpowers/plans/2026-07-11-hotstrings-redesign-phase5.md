# Phase 5 — Script Kind for Hotstrings

## Context

Phase 5 (final phase) of the approved hotstrings redesign (`docs/superpowers/specs/2026-07-07-hotstrings-redesign-design.md`, §§4-9, D8). Phases 1-4 (Text/DateTime/Macro kinds, options, window context) are merged. `HotstringKind.Script` (=3) already exists in the enum on both backend and frontend/CLI mirrors, but it's gated out everywhere: three backend validators reject it (`Must(k => k is Text or DateTime or Macro)`), the emitter has no Script case, and the UI kind selector never offers it. Script content needs **no new persisted field** — it reuses the existing `Replacement` column/DTO field (confirmed: `HotstringDefinition`, `HotstringDto`, `HotstringSnapshot` already carry it generically). This makes Phase 5 lighter than Phase 4: no migration, no new snapshot/restore/revert threading.

**Brainstorm decisions (record in spec as D11-D15 in step 0):**
- **D11** — Script reuses `Replacement`; no new domain/DTO/snapshot columns, no migration.
- **D12** — Brace-balance check is naive `{`/`}` character counting (no string-literal awareness) — consistent with D8's "not a script IDE, no AHK syntax validation" boundary. Known limitation: a `{` inside a quoted string (e.g. `SendText "{"`) counts toward the balance, false-positive-rejecting an otherwise valid script. Documented, not fixed. A string/comment-aware lexical scanner was considered (review pushback) but rejected: it crosses D8's boundary and introduces its own edge cases (escaped quotes, continuation sections). See Unresolved question — user may still choose to widen this.
- **D13** — CLI table Replacement-summary switch gets a Script first-line branch only. Macro rows keep the **raw replacement as-is** decided in Phase 3 (`2026-07-10-hotstrings-redesign-phase3.md` step 17) — no Macro token-summary in Phase 5. Reverting that Phase-3 decision would (a) require the CLI to reference the Application layer for `MacroTokenParser` (the CLI project has zero project references today and would pull FluentValidation/Ardalis into a lightweight tool) or duplicate the parser, and (b) is undefined scope. Deferred to a future CLI-display follow-up.
- **D14** — Grid + mobile kind-filter dropdowns gain Macro and Script items (pre-existing gap: only Text/DateTime were listed) — same edit block as the Script UI work.
- **D15** — Phase 5 "docs note" = CHANGELOG entry + new `docs/cli/hotstrings.md` per-kind CLI reference doc (first one created; prior phases only did CHANGELOG entries).

**Key design facts:**
- Verbatim example (spec §7 ex. 7): Kind=Script, trigger `~ver`, body `MsgBox A_AhkVersion` → `:*:~ver::` + brace-body `{ MsgBox A_AhkVersion }`, warning-badged. This is the golden-test target.
- `HotstringEmitter.BuildOptions` already excludes `T` for Script correctly (comment already lists Script among brace-body kinds) — no change needed there, just confirm with a test.
- Script validation errors surface inline **only on the preview path** today (`ApplyPreviewResult` → `_previewReplacementError`, keyed off `Replacement`). The **Save path does not** — `SaveAsync` routes API validation failures into the generic bottom-of-dialog `_error` alert (with technical `Input.Replacement`-style prefixes). Task 4 extracts the field-error mapping into a reusable method and calls it from Save too, so a save-time brace-imbalance also lands inline on the Replacement field.
- CLI: `HotstringDto`, `HotstringKind`, `KindLabel` already Script-ready; only `FormatReplacementColumn` needs a new Script branch (Macro branch unchanged — see D13).
- Warning copy (spec §6, exact): *"Runs arbitrary AutoHotkey code in the generated script. A syntax error here can break the whole profile script."* — persistent (non-dismissible) `MudAlert Severity.Warning`, unlike the dismissible macro-suggestion alert it's modeled after.

## Task 2 — Emitter goldens (TDD)
- `HotstringEmitter.cs` `BuildBody` switch: add `HotstringKind.Script => BuildScriptBody(hs)`. New `BuildScriptBody(hs)`: wrap `hs.Replacement` in the same outer brace-body shape `BuildMacroBody` uses (`\n{...}\n}`) with **no tokenization and no character escaping** — raw passthrough of the body content.
  - **Verbatim = exact byte preservation of `Replacement` between the braces.** Define precisely (goldens must cover all four): (a) line endings inside the body are preserved as authored (do not normalize `\r\n`↔`\n`); (b) each body line's leading indentation is preserved as-is (the emitter does **not** re-indent per line — the "structural indentation" is only the single wrapper level applied to the opening/closing braces, matching `BuildMacroBody`); (c) leading blank lines in the body are preserved; (d) trailing blank lines in the body are preserved. No trimming.
- Goldens in `tests/AHKFlowApp.Application.Tests/Services/AhkScriptGeneratorTests.cs` (or `HotstringEmitterTests.cs` if that's where Macro's live): `Emit_ScriptKind_WrapsBodyVerbatimInBraces` (exact match to spec §7 ex. 7), `Emit_ScriptKindMultilineBody_PreservesLinesVerbatim`, `Emit_ScriptKindBody_PreservesLeadingIndentation`, `Emit_ScriptKindBody_PreservesLeadingAndTrailingBlankLines`, `Emit_ScriptKind_NeverEmitsTOption`, `Emit_ScriptKindWithContext_WrapsInHotIfAroundBraceBody` (Script + Phase 4 context combo — confirm no interaction bug).

## Task 4 — Frontend dialog
- `HotstringEditDialog.razor`: 4th Script `MudToggleItem` in the kind-selector `MudToggleGroup`. **MudBlazor 9.3's `MudToggleItem` has no `StartIcon`/leading-icon parameter** (only `Text`, plus `SelectedIcon`/`UnselectedIcon` for the checkmark, and `ChildContent` as `RenderFragment<bool>`) — verified via MudMCP. Render the warning affordance via `ChildContent`: a `MudStack Row` with a small warning-colored `MudIcon` (`Icons.Material.Filled.Warning`) + the text "Script", not a nonexistent `StartIcon` attribute. Give the icon an accessible label/tooltip (don't signal "advanced/danger" by color alone).
- **Kind-switch transition matrix** (`OnKindChangedAsync`) — resolves the keep-vs-clear question explicitly; Text/Macro/Script all use the same raw `Replacement`, so **never clear it when moving among those three**:
  - **Text ↔ Macro ↔ Script**: always preserve `Replacement`. Add a Script arm modeled on the existing Text/Macro arms (which already only clear the DateTime-specific fields, never `Replacement`). A confirmation dialog is allowed **only** to explain a changed *interpretation* of the same text (as Macro→Text already does for `{{...}}` tokens) — e.g. Text/Macro→Script "your text will run as AutoHotkey code", Script→Text "your code will be typed as literal text" — but the confirm must never precede clearing shared raw content; on confirm, only `Item.Kind` changes.
  - **→ DateTime**: keep the existing confirm-and-clear behavior (DateTime has no reusable `Replacement`).
- Persistent (non-dismissible) `MudAlert Severity="Severity.Warning"` shown unconditionally when `Item.Kind == HotstringKind.Script`, with the exact copy from spec §6. Model structurally on the existing macro-suggestion alert but drop the dismiss button/dismissal state.
- **"Use Script?" suggestion (spec §54; Phase 3 deferred it here):** when `Item.Kind == HotstringKind.Text` and the Replacement "looks like AHK code" (lines starting with `Send`/`::`/`#`, or contains braces — a small frontend helper mirroring the existing `MacroTokens` detector), show a **dismissible** `MudAlert Severity="Severity.Info"` (`data-test="script-suggestion"`) with copy "This looks like AutoHotkey code — switch to Script?" and a "Switch to Script" action calling `OnKindChangedAsync(HotstringKind.Script)`. Per-dialog-instance dismissal, exactly like the macro suggestion. Accessible text, not color-only.
- **Script editor sizing (monospace alone is not enough):** apply a monospace styling hook (new CSS class, e.g. `ahk-mono`) when `Item.Kind == HotstringKind.Script`, **and** give the Replacement `MudTextField` a taller editing surface for Script — `Lines="8"` (or `AutoGrow` with `MaxLines` bounded, ~8-16) instead of the default `Lines="3"` — and disable spellcheck (`spellcheck="false"` user-attribute) since it's code. Verify no horizontal overflow on a narrow (phone) viewport during E2E.
- **Save-time inline errors:** extract the field-error mapping currently inline in `ApplyPreviewResult` (the `Trigger`/`Replacement`/`ContextValue` dotted-key switch) into a reusable method, and call it from `SaveAsync`'s failure branch too, so a validation failure on Save (e.g. brace imbalance) lands on the Replacement field inline instead of the generic `_error` alert. Keep the existing `Conflict`→`_triggerError` behavior; only the `Validation` status routes through the shared mapper.
- Replacement field: (monospace hook covered above).
- `HotstringEditModel.cs`: new `ScriptSummary` computed property (first line of `Replacement`, trimmed/ellipsized) analogous to `DateTimeSummary`/`ContextSummary`, for grid/mobile reuse.
- bUnit: `KindToggle_SelectScript_ShowsPersistentWarningAlert`, `KindToggle_SelectScript_AppliesMonospaceClass`, `KindToggle_TextToScript_PreservesReplacement`, `KindToggle_ScriptToText_PreservesReplacement`, `ScriptSuggestion_TextLooksLikeCode_ShowsDismissibleAlert`, `Save_ScriptKind_SendsReplacementVerbatimInCreateDto`, `PreviewError_ScriptBraceImbalance_ShownInlineOnReplacementField`, `Save_ScriptBraceImbalance_WithPreviewCollapsed_ShowsInlineReplacementError`, `ScriptSummary_MultilineBody_ReturnsFirstLineOnly`.
