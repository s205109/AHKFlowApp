## Goal

Publish the AHKFlow CLI to the Windows Package Manager (`winget`) community repository so that Windows users can install it with `winget install AHKFlow.CLI` and get `ahkflow.exe` on PATH. The Winget package wraps the existing `ahkflow-win-x64.zip` released for backlog 030 — no new build artifact.

The first submission targets the existing v0.1.1 GitHub Release. CI automation of subsequent versions is deferred to a later backlog item (032).

## Decisions

- **PackageIdentifier:** `AHKFlow.CLI`. Short, matches the executable name. Fallback if moderators reject the unverified `AHKFlow` publisher: `Segocom.AHKFlowCLI`.
- **Submission tooling:** `wingetcreate` (Microsoft-supported). Run locally, not in CI.
- **Submission approach:** manual one-shot for v0.1.1. Document the steps so the next release can re-run them. CI automation is out of scope.
- **Installer type:** `InstallerType: zip` with `NestedInstallerType: portable`. Winget handles PATH registration via its symlink directory — no MSI, no MSIX, no custom installer.
- **Source binary:** the existing GitHub Release asset `https://github.com/<owner>/AHKFlowApp/releases/download/v0.1.1/ahkflow-win-x64.zip`. No rebuild.
- **Signing:** unsigned for v1. SmartScreen will warn on first launch — acceptable for a 0.x CLI. Re-evaluate when a code-signing cert is available.
- **Version policy:** submit v0.1.1 as-is. No pre-bump to v1.0.0.
- **License metadata:** MIT, sourced from the existing `LICENSE` file at repo root. License URL points to `https://github.com/<owner>/AHKFlowApp/blob/main/LICENSE`.
- **Runbook location:** `docs/cli/winget-submission.md`, co-located with the existing zip install doc.

## Current State

## Manifest Design

Three Winget manifests, generated by `wingetcreate new` and reviewed by hand:

**`AHKFlow.CLI.installer.yaml`** (key fields):

```yaml
PackageIdentifier: AHKFlow.CLI
PackageVersion: 0.1.1
InstallerType: zip
NestedInstallerType: portable
NestedInstallerFiles:
  - RelativeFilePath: ahkflow.exe
    PortableCommandAlias: ahkflow
Installers:
  - Architecture: x64
    InstallerUrl: https://github.com/<owner>/AHKFlowApp/releases/download/v0.1.1/ahkflow-win-x64.zip
    InstallerSha256: <sha256 of the released zip>
Commands:
  - ahkflow
ManifestType: installer
ManifestVersion: 1.6.0
```

`Scope` is intentionally omitted — `winget validate` warns that it is not
supported for `InstallerType: portable`.

**`AHKFlow.CLI.locale.en-US.yaml`** (key fields):

```yaml
PackageIdentifier: AHKFlow.CLI
PackageVersion: 0.1.1
PackageLocale: en-US
Publisher: AHKFlow
PackageName: AHKFlow CLI
License: MIT
LicenseUrl: https://github.com/<owner>/AHKFlowApp/blob/main/LICENSE
ShortDescription: Command-line tool for managing AutoHotkey hotstrings and hotkeys.
PackageUrl: https://github.com/<owner>/AHKFlowApp
ReleaseNotesUrl: https://github.com/<owner>/AHKFlowApp/releases/tag/v0.1.1
Tags:
  - autohotkey
  - hotstring
  - cli
  - windows
ManifestType: defaultLocale
ManifestVersion: 1.6.0
```

**`AHKFlow.CLI.yaml`** (version manifest): `PackageIdentifier`, `PackageVersion: 0.1.1`, `DefaultLocale: en-US`, `ManifestType: version`, `ManifestVersion: 1.6.0`.

`portable` NestedInstallerType is the key choice: Winget extracts the zip to `%LOCALAPPDATA%\Microsoft\WinGet\Packages\<id>_...\` and creates a symlink for `ahkflow` in `%LOCALAPPDATA%\Microsoft\WinGet\Links` (already on PATH for the user scope). Users do not need to edit PATH manually — that's the whole win over the zip flow.

## Submission Flow

1. **Compute SHA256** of the v0.1.1 release zip:
   `Get-FileHash -Algorithm SHA256 ahkflow-win-x64.zip` after downloading from GitHub Releases.
2. **Generate manifests:** `wingetcreate new <installer-url>` and fill in publisher/license/description interactively, or `wingetcreate new-version` if a draft exists.
3. **Validate locally:** `winget validate <manifest-dir>` exits 0; `winget install --manifest <manifest-dir>` succeeds on a clean profile.
4. **Smoke-test installed binary:** new shell, `ahkflow --help`, `ahkflow login`, `ahkflow hotstring list`, `ahkflow logout` — all against prod, no `AHKFLOW_*` overrides.
5. **Submit PR:** `wingetcreate submit --token <gh-pat> <manifest-dir>` (or fork `microsoft/winget-pkgs` + manual PR). Address moderator feedback.
6. **Post-merge verification:** `winget search AHKFlow.CLI` returns the entry; `winget install AHKFlow.CLI` (no `--manifest`) installs from the community feed; `winget uninstall AHKFlow.CLI` removes it cleanly.

## Out of Scope

- CI automation for Winget submissions (separate backlog item 032).
- Code signing the `ahkflow.exe` binary.
- MSI / MSIX installer formats.
- Auto-update beyond what `winget upgrade` provides natively.
- macOS / Linux package managers.
- Changes to CLI command surface, runtime configuration, or auth flow.
- Changes to `release-cli.yml` or `scripts/publish-cli.ps1`.

## Risks and Mitigations

- **Publisher rejection** — `AHKFlow` is not a verified Winget publisher. If moderators object, rename `PackageIdentifier` to `Segocom.AHKFlowCLI` and resubmit. This is a manifest-only change, no rebuild.
- **SmartScreen warning** — unsigned binary triggers a warning on first launch. Document this in `windows-install.md` so users know to expect it.
- **SHA256 drift** — the GitHub Release asset must not be re-uploaded after manifest submission, or the SHA changes and Winget install fails. Mitigation: submit only against immutable tag releases, never against `latest` or rolling tags.
